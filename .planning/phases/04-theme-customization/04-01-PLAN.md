<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| User input → CSS variable | The custom font name input (D-02, implemented in Plan 02) passes user-typed string into `document.documentElement.style.setProperty('--font-sans', value)` |
| localStorage → state | Font/density preferences read from localStorage on init; could be tampered by a malicious page if webview is shared (not applicable in Wails desktop app) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-01 | Tampering | Google Fonts @import removed | accept | Google Fonts CDN removal eliminates a potential MITM attack vector against the font CDN; woff2 files are static assets served locally — no network trust boundary |
| T-04-02 | Information Disclosure | localStorage keys (cmdex-ui-font, cmdex-mono-font, cmdex-density) | accept | Values are non-sensitive UI preferences (font names, density labels); no PII; desktop app — localStorage is per-profile, not shared cross-origin |
| T-04-03 | Spoofing | Custom font name injected as CSS property value | accept | `style.setProperty('--font-sans', value)` sets a CSS property, not innerHTML — CSS property injection cannot execute JavaScript; Wails webview has no cross-origin page that could exploit this; value is used only for font-family rendering |
| T-04-04 | Denial of Service | Invalid woff2 file causes font rendering failure | accept | Fallback chain in @font-face (`system-ui, -apple-system, sans-serif`) ensures readable text even if a bundled font fails to load; app functionality is unaffected |
</threat_model>
