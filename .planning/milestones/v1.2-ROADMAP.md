# Milestone v1.2: DB Migration Refactor

**Status:** Planning
**Phases:** 8-9 (2 phases)
**Total Requirements:** 12

## Overview

Replace the monolithic inline `migrate()` function in `db.go` with a per-file up/down migration pattern. Each of the 9 existing migrations moves to its own numbered Go file in a `migrations/` package. A runner in `db.go` handles discovery, ordering, transaction wrapping, and idempotency. Existing databases upgrade seamlessly with no data loss and no user-visible change.

- [ ] **Phase 8: Migration Package** - Create migrations/ package with Up/Down interface, port all 9 migrations

## Phase Details

### Phase 8: Migration Package

**Goal**: A `migrations/` package exists with all 9 existing migrations ported to individual numbered files, each with `Up` and `Down` functions

**Depends on**: Nothing (first phase of milestone)

**Requirements**: MIGR-01, MIGR-02, MIGR-03, MIGR-04, ROLL-01

**Success Criteria** (what must be TRUE):
1. A `migrations/` directory exists containing Go files named `0001_initial.go` through `0009_*.go`
2. Every migration file compiles and exports `Up(tx *sql.Tx) error` and `Down(tx *sql.Tx) error` functions
3. The SQL inside each `Up` function matches the corresponding inline migration block from the original `migrate()` in `db.go`
4. Each `Down` function contains the inverse SQL needed to undo that migration step
5. Running `go build ./...` with only Phase 8 changes produces no compile errors

### Phase 9: Runner & Integration

**Goal**: The migration runner in `db.go` automatically applies pending migrations in order, and the app builds and runs identically against both fresh and existing databases

**Depends on**: Phase 8

**Requirements**: RUN-01, RUN-02, RUN-03, RUN-04, ROLL-02, COMPAT-01, COMPAT-02

**Success Criteria** (what must be TRUE):
1. Starting the app against a fresh database applies all 9 migrations in numeric order and the app reaches the main UI without errors
2. Starting the app against an existing database with migrations already applied skips all applied steps (idempotent — no duplicate execution)
3. Each migration step executes inside its own transaction; a failure in step N rolls back step N without affecting steps 1 through N-1
4. Calling `DB.RollbackTo(version int)` steps the schema back to the target version using each migration's `Down` function
5. The original monolithic `migrate()` function is fully removed from `db.go` with no remaining references

## Milestone Summary

### Key Decisions

- Custom pure-Go implementation — no external migration tools (Goose, Atlas, golang-migrate)
- Go files only — no SQL files, consistent with existing codebase convention
- `Down()` functions required on every migration file (not optional) to support `RollbackTo`
- Runner tracks state via existing `schema_version` table — no new tracking table needed
- `RollbackTo` is a dev/testing utility; it is not exposed to the frontend

### Technical Notes

- Migration discovery: package-level `var Migrations []Migration` slice ordered by file prefix
- Transaction wrapping: runner calls `db.conn.Begin()` per migration step, not per batch
- Existing `schemaVersion` constant in `db.go` updated to reflect total migration count
- After runner replaces `migrate()`, FTS rebuild trigger (`INSERT INTO commands_fts(commands_fts) VALUES('rebuild')`) must be preserved in the appropriate migration file
